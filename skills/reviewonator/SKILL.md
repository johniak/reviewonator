# Review a pull request with Reviewonator

Accept one GitHub pull request URL as the argument. Require `gh` and the `reviewonator` executable.
Review language configuration: write public pull request comments and the review summary in English; write private reviewer explanations in English.

## Run the workflow

1. Verify prerequisites with `gh --version`, `gh auth status`, and `reviewonator --help`. Only in Codex on macOS, never in Claude Code, a sandboxed `gh auth status` can fail because the sandbox cannot read GitHub CLI credentials from Keychain. In that specific environment, retry the check outside the sandbox with escalated permissions before concluding that authentication is invalid. Stop with a direct installation or authentication instruction only when the host-level check also fails. Never run `gh auth login`, `gh auth logout`, or print, copy, export, or persist the token as a workaround.
2. Read the repository's `CLAUDE.md`, `AGENTS.md`, and relevant nested instruction files. Follow any repository knowledge or memory files referenced by those instructions. Apply the closest nested rules to each changed file. If the repository has a review skill such as `.claude/skills/review/SKILL.md` or `.agents/skills/review/SKILL.md`, read it fully. Its review process, file-coverage requirements, domain expertise, checklist, and standards are authoritative; follow them with the same depth as a direct invocation. This skill overrides only the final output shape, Reviewonator UI handoff, and publishing workflow. Do not weaken, shorten, or suppress the repository skill's verified findings to fit this workflow.
3. Load PR metadata, changed files, existing discussion, reviews, and inline review threads with `gh`. Only in Codex on macOS, never in Claude Code, when Keychain credentials are unavailable in the sandbox, request escalated execution for authenticated `gh` commands so they run with the user's existing host credentials. Prefer narrowly scoped, read-only command prefixes and group compatible reads to avoid unnecessary approval prompts. Do not introduce `GH_TOKEN` or `GITHUB_TOKEN` into the agent environment. Load the final combined base-to-head diff with `gh pr diff <PR_URL> --color never`. Do not use `--patch`: it emits a per-commit mail series containing intermediate revisions, which makes final line validation unreliable. Read the title, description, and discussion first to establish the PR's goal; every finding must be relevant to that goal.
4. Use an isolated worktree or temporary clone when full repository context is unavailable. Never modify the user's active working tree during review.
5. Scan every changed file and every changed line, then read the complete final version of every changed source, test, migration, configuration, and fixture file. Read generated files, lockfiles, and snapshots to the depth needed to verify their changes. Trace changed functions through every existing and new caller, plus models, permissions, tasks, signals, lifecycle hooks, cascade behavior, configuration, templates, and tests; do not stop at the edited function. When code is extracted into a shared helper or an existing caller is rerouted, compare that caller's base and head behavior explicitly, including exception propagation, rollback, ordering, return values, and side effects.
6. Build a private review ledger for every changed file. Record the changed entry points, old and new side effects, external or asynchronous boundaries, data sent across each boundary, feature-flag states, tests that exercise the real production path, applicable repository rules, and every candidate concern. Give each candidate a disposition: verified finding, already reported, pre-existing, disproved, or unverified, with the evidence that justifies that disposition. Do not stop after finding the first serious issue; finish the ledger and every mandatory review pass before producing JSON. A stronger finding, an already-blocking verdict, or a candidate's lower severity is never a reason to discard a separate verified actionable issue. Use the ledger for reasoning; do not include it in the review JSON.
7. Perform the mandatory adversarial passes in **Risk analysis** below. Review only behavior introduced or modified by the PR, but use untouched code to prove consequences and call paths.
8. Before writing each finding, verify the exact offending line against the final combined base-to-head diff or `git blame`, then verify the affected call sites and contracts. Drop the finding if it predates the PR or cannot be proven. For a problem caused by a new path bypassing old code, anchor the comment to the added line that introduces or selects the new path, not to the untouched code that became dead.
9. Check issue comments, submitted reviews, and inline review threads, including their resolved or outdated state. Do not repeat an issue already raised.
10. Write every verified, actionable, non-duplicate finding that the authoritative review process would report to the review JSON described in [references/review-schema.md](references/review-schema.md). New findings start pending so the human, not the agent, decides whether to include, revise, or reject them; decisions carried from an earlier revision round retain their state. Combine findings only when they describe the same defect, consequence, fix, and best diff anchor; shared root cause alone is not enough. Sort findings by severity and impact before assigning stable IDs.
11. Copy both language names literally from **Review language configuration** into `languages`; never infer or restore defaults. Write public PR-author comments and the review summary in the configured comment language. Write separate private reviewer explanations in the configured reviewer-note language. Before launching, inspect the completed JSON and correct it unless `languages.comments`, `languages.reviewerNotes`, every `body`, `summary`, and `reviewerExplanation` follow those exact settings.
12. Run `reviewonator <PR_URL> --review-file <JSON_PATH>` in the foreground and wait for its final JSON result. Only in Codex on macOS, never in Claude Code, request escalated execution outside the sandbox when needed: Reviewonator's child `gh` process must be able to read the user's Keychain credentials, bind its loopback server, and open the browser. This execution approval does not authorize publication; explicit confirmation in the Reviewonator UI remains mandatory. The ready URL means the human review has started, not that the command completed. While the UI is open, do not dump the verdict, findings, or a second review summary into chat; the UI is the review surface.
13. Handle the result:
    - `revision_requested`: preserve every unaffected finding and its stable ID. Treat `selectedCommentIds` and `rejectedCommentIds` as the human's carried decisions. On each surviving comment, set `included: true` only when its ID is selected, set `rejected: true` only when its ID is rejected, and omit both flags for pending or revision-requested comments. Never set both flags. New comments start pending. If a decided finding is disproved and removed, do not keep a placeholder merely to preserve its state. For each item in `requests`, re-verify and update the requested finding, including its canonical comment and private explanation when relevant. For each item in `newComments`, inspect the specified file, side, line, surrounding code, and the user's proposed point. Verify the concern instead of trusting it blindly, then turn each valid concern into a new finding with a stable ID, accurate severity and anchor, a concise human-sounding canonical comment, and a proper private reviewer explanation. If the proposed line cannot be published inline, use the nearest accurate added-line anchor or a general finding. Rewrite the JSON and run Reviewonator again.
    - `published`: report the returned GitHub review URL.
    - `cancelled`: report that nothing was published.
14. Remove temporary worktrees, clones, and review files created by this workflow.

## Risk analysis

Complete every applicable pass before choosing a recommendation:

- **Behavior and side effects:** compare the base and head behavior for each changed entry point. Enumerate user-visible and external side effects such as emails, notifications, payments, writes, jobs, and API calls. For migrations and parallel-run implementations, determine whether the new path replaces, duplicates, or races the old path. Treat duplicate delivery as a real operational consequence unless the PR explicitly defines and tests it as intentional.
- **Existing-caller regression:** when a PR extracts, centralizes, wraps, or reroutes behavior, enumerate every old and new caller and compare its base and head contract. Check whether caught exceptions used to escape, whether an atomic operation used to roll back, whether side effects moved across a transaction boundary, and whether callers now receive different results. Test the existing flows as well as the new one; a working new entry point does not prove the refactor preserved old behavior.
- **Framework lifecycle and cardinality:** trace implicit behavior triggered by the changed operation: ORM cascades, signals, hooks, middleware, model methods, callbacks, cleanup handlers, and framework events. Estimate how often each hook runs for realistic collection sizes and detect per-row queries, network calls, tasks, or duplicate cleanup hidden behind a single high-level call. Pay special attention to external work performed while database locks or transactions remain open.
- **Trust boundaries and data flow:** for every new or changed API, queue, broker, webhook, analytics, storage, logging, email, or third-party integration, trace the payload field by field from source to destination. Look for secrets, tokens, passwords, PII, rendered bodies, attachments, identifiers, and data that becomes durable in a broker or external system. Serialization safety is not confidentiality or authorization.
- **Enabled-path analysis:** inspect the fully enabled and fully configured production path. A default-off feature flag, allowlist, fallback, exception wrapper, or successful legacy path limits present blast radius but does not prove the enabled behavior is correct or safe.
- **Failure and retry analysis:** check partial failure, retries, idempotency, ordering, transaction boundaries, stale data, and what happens after an external system accepts work but the caller sees a timeout.
- **Test validity:** identify which production path each test actually exercises. Coverage of helpers, stubs, obsolete dispatch tables, or mocked indirection does not prove the real call path. Look for dead code kept alive only by tests and missing assertions at changed boundaries.
- **Repository conventions:** turn every applicable requirement from `CLAUDE.md`, `AGENTS.md`, nested instructions, and the repository review skill into a checklist for the changed files. Verify test structure, fixture and factory use, transaction markers, mocking policy, query-count requirements, coverage rules, logging, and other explicit conventions. Treat a confirmed violation as a finding unless the PR documents a valid exemption; passing tests do not override an authoritative project rule.
- **Dependency contracts:** for every changed use of a third-party library, framework, SDK, or tool, identify the exact installed version from the manifest or lockfile and consult its official documentation for that version before judging the behavior. If the documentation is ambiguous, inspect the upstream source, type definitions, or release notes. Do not rely on remembered API behavior.
- **Abstraction cost:** check whether new layers, protocols, registries, and wrappers have multiple real consumers or merely speculative flexibility. Report this only when the added complexity has a concrete maintenance or correctness cost.

## Review rules

- Act as a critical, skeptical senior software engineer and security-focused reviewer. Challenge happy-path assumptions and actively look for concrete failure modes, but never manufacture findings merely to appear thorough.
- Report actionable problems only. Do not add praise or filler.
- Do not use the PR description, happy-path tests, broad coverage, a disabled-by-default flag, or swallowed exceptions as evidence that a risky path is correct. Verify the path itself.
- Target inline comments only at added lines in the diff. A finding about a specific file or function must be inline on the narrowest relevant added line. Use a general comment only for a genuinely cross-cutting issue or when no added line is an accurate target; never hide a file-specific finding in a general comment.
- Explain the concrete consequence and the necessary change in each comment.
- Keep each public comment compact: normally 2–4 sentences and at most about 80 words, excluding a small code suggestion. State the problem, consequence, and required change once; omit review-process narration and repeated evidence. Go longer only when shortening it would make the fix ambiguous.
- Be specific. When proposing a concrete replacement, include a concise code suggestion when it makes the fix clearer.
- Write `body` and `summary` naturally in the configured comment language. Write `reviewerExplanation` in the configured reviewer-note language using natural equivalents of `What: ... Why: ...`; explain the issue instead of merely translating `body`.
- Write every canonical comment as a thoughtful human teammate would write it: direct, specific, natural, and proportionate to the issue. Remove robotic scaffolding, review-process narration, formulaic transitions, hedging filler, severity labels, and finding IDs from the canonical sentence itself. Read each comment once as dialogue before finalizing it and rewrite anything that sounds generated, templated, or like a linter message.
- Report all verified actionable findings, including non-blocking suggestions and nits that satisfy the authoritative repository review rules. There is no preferred, minimum, or maximum finding count: never optimize for fewer comments and never add filler to produce more. Quality, verification, and usefulness decide whether a finding exists. Reject speculation, but never use brevity or the existence of a stronger finding as a reason to omit a separate verified concern; the human decides whether to include, revise, or reject each finding in the UI.
- Before producing JSON, audit every ledger disposition once more. Every `verified finding` must appear in JSON unless it is an exact duplicate of an existing PR comment, and every dropped candidate must have concrete evidence for `already reported`, `pre-existing`, `disproved`, or `unverified`. Do not silently downgrade a behavioral regression into a nearby documentation or style comment.
- `summary` is always optional. When a useful overall message exists, keep it concise and problem-oriented, led by the highest-impact unresolved risk. Do not invent a summary, praise the implementation, narrate how much code you read, or list reasons the PR is probably safe. Use an empty string whenever no separate overall message adds value, regardless of the recommendation.
- Sort findings by severity (`security`, `bug`, `warning`, `suggestion`, `nit`), then by blast radius and impact. Use `S1`, `S2`, ... for inline findings and `G1`, `G2`, ... for general findings; keep IDs stable across revision rounds.
- Classify severity from the proven consequence, not from likelihood or scope alone. Unauthorized destructive behavior, deletion or corruption of data, or a race that can persist invalid state is a `bug` (or `security` when it crosses an authorization boundary), not a `warning`. Recommend `REQUEST_CHANGES` whenever such a defect remains unresolved.
- Run relevant tests or focused checks when they can verify a concern without changing external state.
- Never publish with `gh pr review` or `gh api` from the skill. Reviewonator owns publication and requires explicit user confirmation in its UI.
- Never manufacture a summary merely to justify `COMMENT`, `APPROVE`, or `REQUEST_CHANGES`.
- After the review, update durable project memory only when the repository instructions explicitly require it; never change the reviewed PR branch to do so.

## Severity

- `security`: exploitable security, authorization, privacy, or secret exposure issue.
- `bug`: behavior is incorrect or data can be corrupted or lost.
- `warning`: material reliability, performance, compatibility, or maintainability risk.
- `suggestion`: concrete improvement that is useful but not required for correctness.
- `nit`: localized readability or consistency issue worth the author's attention.

Recommend `REQUEST_CHANGES` for unresolved security issues or material bugs, `COMMENT` for non-blocking findings, and `APPROVE` when no blocking findings remain.
